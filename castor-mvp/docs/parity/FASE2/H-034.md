# H-034 — Botón "+ Pago" inline en cada fila · P0 (CRÍTICO)

**Módulo:** Ventas → tabla → última columna · **Prioridad:** P0 (ventaja funcional clave)

## Diagnóstico
Demo6 (`Castor_Dashboard_Demo6.html:5322,5334`) tiene un botón "+ Pago" por fila que registra un
abono sin abrir el detalle. React no lo tenía (solo desde el detalle).

## Cambio
`src/views/Ventas.jsx` — agregada columna de acciones (header vacío, `sortable:false`) con botón
"+ Pago" que hace `e.stopPropagation()` + abre el modal de pago existente
(`setPay({ orderId, amount:'', method:'Transferencia', bankId: bankAccounts[0]?.id })`).
**Reutiliza `savePayment`** (DRY): crea el registro `PAYMENTS`, recalcula `order.paid` y genera el
asiento de cobro. Saldo + barra PAGO se actualizan en vivo (las filas derivan de `orders`).

## Verificación funcional (Preview MCP, :5173) — P-1253
1. Baseline: total $3.450.000, saldo $1.725.000, pago 50%, botón "+ Pago" presente.
2. Clic "+ Pago" → modal "Registrar pago" abre con el pedido precargado.
3. Registrar $1.725.000 → se crea `PAG-505`; `paid` recalculado a 50% (ver caveat).
4. Segundo abono $862.500 → **pago 75%, saldo $862.500** (recálculo `2.587.500/3.450.000`). ✅ saldo+barra actualizan en vivo.
- `stopPropagation` evita abrir el detalle al pulsar el botón. eslint exit 0. Datos de prueba reseteados (efímeros).

## Edge / caveats
- **[DESCUBIERTO EN VALIDACIÓN] EX-F2-02**: el `paid%` del seed no siempre está respaldado por
  registros `PAYMENTS`; `savePayment` recalcula `paid` desde la suma de registros, por lo que el primer
  abono puede "saltar" el % al valor derivado. NO es bug crítico (sin crash/pérdida), es **preexistente**
  (misma `savePayment` del botón del detalle). Registrado en `HALLAZGOS_EXTRA.md`, no se toca.
