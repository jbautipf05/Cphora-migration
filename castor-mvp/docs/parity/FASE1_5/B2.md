# B2 — Identidad de cliente: `customerId` (FK) canónico (Fase 1.5)

**Tipo:** corrección técnica (modelo de datos) · **Severidad:** 🔴 crítica · **Origen:** REVISION_TECNICA_PRE_FASE2 §BLOQUEANTES / TD-29 · **ADR:** [ADR-011](../../adr/ADR-011-identidad-cliente.md).

## Problema
Las relaciones Cliente↔Pedido/Cotización se hacían por **nombre** (`clientName`), porque los
pedidos creados en runtime hardcodeaban `customerId: null` (`Ventas.jsx`, `Auditoria.jsx`).
Renombrar un cliente **desvinculaba su historial en silencio**; además el motor contable
recibía un nombre como id de tercero. (El seed sí traía `customerId`; el modelo ya era FK-ready.)

## Decisiones (OK del usuario, 2026-05-26)
Opción A del ADR-011: `customerId` FK canónico + `clientName` como denormalización de display ·
walk-in **crea cliente** automáticamente al crear pedido · display **refleja** el nombre actual ·
placeholders de ADR-010 renumerados (012/013/014).

### Sub-fix obligatorio: `nextId` (CR-09 / TD-30) — descubierto durante B2
`saveOrder` pasó a llamar `nextId` **dos veces** (PED + C). Se detectó en verificación que el
**segundo** `nextId` devolvía `undefined` (cliente creado sin `id`, pedido con `customerId`
undefined). Causa: `nextId` retornaba el id desde **dentro del updater de `setState`**, que React
solo evalúa síncronamente para el primer `setState` del tick. **Se reportó como bug crítico y, con
OK del usuario, se arregló de raíz**: contador espejado en `useRef` con bump síncrono. Esto también
corrige el **bug latente pre-existente** del modal de cotización (autocrear lead `nextId('L')` +
`nextId('COT')`).
