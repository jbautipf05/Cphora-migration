# REG-H035-06 — Validaciones del Bloque Cliente (modal Nueva Venta)

**Fase:** 2.5.2 · **Prioridad:** P0 · **Bloque:** B-básico (1º hallazgo)
**Archivo:** `src/components/NuevaVentaModal.jsx`
**Estado:** ✅ Implementado y validado E2E (0 errores de consola).

## 1. Síntoma (regresión de Fase 2)

El modal aceptaba registrar una venta con **todos los campos del cliente vacíos**.
`handleSubmit` sólo comprobaba `custName` y la existencia de ≥1 ítem; no había
validación funcional ni mensajería de error, pese a que H-035 se cerró como 23/23.

## 2. Inspección de Demo6 (fuente de verdad)

`openSaleForm` (`Castor_Dashboard_Demo6.html:5543-5601`) y `saveSale` (`:5975-5985`).

**Mecanismo real de Demo6 = HTML5 nativo**, no validación JS custom:

- Los **11 campos** del bloque cliente llevan `required` nativo:
  `custName, custDoc, custPhone, custEmail, custCity, custAddress,
  custDeliveryAddress, medio, pdv, cierreVenta, abonoBancos`.
- `custEmail` tiene además `type="email"` → el navegador valida el formato.
- **`custPhone` sólo lleva `required`** — Demo6 NO valida formato de teléfono.
- **No hay `minlength`** en ningún campo.
- La validación dispara **on-submit** (el `<form onsubmit>` deja que el navegador
  bloquee antes de llamar a `saveSale`); el asterisco vive sólo en el texto del label.
- `saveSale` repite en JS sólo `custName` (+ `docType` + `paySoporte`) como backstop.

## 3. Decisión de implementación (fiel + consistente)

El modal React no es un `<form>` nativo con `required` (usa componentes controlados
`Input`/`Select` y un botón `onClick`), así que se replica el **conjunto obligatorio**
de Demo6 con el patrón inline ya establecido en el proyecto — **H-004 de `Leads.jsx`**
(`validate()` → objeto `errs` → toast + mensaje rojo inline; on-submit; se limpia al
editar):

| Aspecto | Demo6 | Implementación React | Nota |
|---|---|---|---|
| 11 campos obligatorios | `required` nativo | bloqueo de submit + inline | **fiel** (mismo set) |
| Disparo | on-submit | on-submit | **fiel** |
| Correo | `type="email"` nativo | `type="email"` + regex `EMAIL_RE` | **fiel** (refuerzo) |
| **Teléfono formato** | **sólo `required`** | sin letras + ≥7 dígitos | **divergencia documentada** |
| Visual error | burbuja nativa del browser | texto rojo inline bajo el campo | enhancement H-004 |
| Toast general | (sólo backstop en saveSale) | `toast('Revisa los campos del cliente marcados','warn')` | enhancement H-004 |

> **Divergencia consciente (teléfono):** Demo6 no chequea formato de celular, pero el
> proyecto React ya adoptó el refuerzo "sin letras + ≥7 dígitos" para el MISMO tipo de
> campo en `Leads.jsx:172-182` (H-004), documentado allí como enhancement deliberado
> sobre el `required` nativo de Demo6. Se aplica aquí por **consistencia interna** del
> código React y porque fue pedido explícitamente al abrir el hallazgo. Si se prefiere
> paridad estricta (sólo `required`), se elimina el bloque de 2 líneas de `validateClient`.

**No se tocó** la lógica de ítems (REG-07), control general (REG-08) ni pago (REG-09):
fuera del alcance de este hallazgo. La guarda mínima preexistente de ítems se conservó.

## 4. Cambios (quirúrgicos)

- `import { useToast } from './Toast'` + `const EMAIL_RE`.
- Estado `errors` + `toast`; `set()` limpia el error del campo al editar (espejo de
  `Leads.jsx:151-154`); `useEffect(open)`, `reset()` y `selectCustomer()` limpian errores.
- `validateClient(v)` → objeto `errs` para los 11 campos (+ formato tel/email).
- `handleSubmit`: valida cliente, setea errores y hace `toast(...)` si hay; si no, sigue.
- `errLine(k)` helper + `{errLine('<campo>')}` bajo cada uno de los 11 campos.
