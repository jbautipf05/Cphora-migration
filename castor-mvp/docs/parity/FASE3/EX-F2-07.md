# EX-F2-07 — La venta reserva stock + registra stockMoves (RESUELTO)

**Contexto:** pre-trabajo de Fase 3 (deuda detectada en auditoría de cierre EX-F2-05). La venta estándar
NO reservaba `finishedStock` ni registraba `stockMoves` para ítems de stock, ni bajaba el `stock` del
producto maestro — a diferencia de Demo6 `saveSale:6126-6145`. Esto es P0 para los módulos visuales de
Fase 3 (Almacén/Inventario/Despacho dependen de esos registros). Lo adelanté mientras se arma el inventario.

## Inspección de Demo6 (`saveSale`)
- **`:6103-6106`** — guard de suficiencia ANTES de crear nada: si `avail < it.qty` para un ítem de stock,
  `toast('Stock insuficiente …')` y `return` (aborta la venta).
- **`:6126-6145`** — por cada ítem `type==='stock'`: descuenta de `finishedStock` disponible (FIFO entre
  bodegas), marca `status:'reservado'` + `orderId`/`pedidoId` (o divide el registro si es consumo parcial),
  baja `products[].stock` (`Math.max(0, …)`), y hace `stockMoves.push({type:'salida_venta', …})`.

## Implementación (fiel a Demo6, atomicidad B2)
1. **Guard de suficiencia** (tras validar contenido, antes de cualquier mutación): para cada ítem de stock,
   `avail = Σ finishedStock(disponible).qty`; si `avail < qty` → toast error + return. Solo venta estándar.
2. **Reserva** tras crear la orden, en **UN `setState` atómico** que recalcula `finishedStock` + `products`
   + `stockMoves` desde el mismo `s` (sin lecturas intermedias):
   - FIFO con `flatMap`: consumo total → registro pasa a `reservado` (con `orderId`/`pedidoId`); consumo
     parcial → registro disponible reducido + **nuevo** registro `reservado` (qty tomada, id `uid('FS')`).
   - `products[].stock = Math.max(0, stock - qty)`.
   - 1 `stockMove` por ítem: `{ type:'salida_venta', orderId, pedidoId, productId, qty, date, reason, by }`.
3. Solo aplica a **venta estándar con ítems `tipo:'stock'`**. Innovación y producción no afectan inventario
   (por construcción: la reserva vive en la rama no-innovación y filtra `tipo==='stock'`).

## Decisión documentada (divergencia con Demo6)
Al consumir un registro **entero**, React conserva su `qty` con `status:'reservado'`; Demo6 lo deja en
`qty:0` (`saveSale:6132-6133`). Se diverge a propósito porque React calcula KPIs de inventario en vivo
(InventarioTerminado/Inicio suman `qty`), donde un reservado `qty:0` haría desaparecer unidades. El total
reservado por venta queda correcto y trazable por `orderId`.
